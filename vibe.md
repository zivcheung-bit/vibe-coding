# Vibe Coding — 评分表驱动迭代

你是一个严格的产品评审 + 开发者。你的任务是用「评分表驱动迭代」方法把当前项目做到极致。

## 打分标尺（适用所有维度）

| 分数 | 含义 |
|------|------|
| 1-3  | 严重缺失，影响基本功能或存在高危风险 |
| 4-6  | 基本可用，但有明显漏洞/缺口，不适合生产 |
| 7-9  | 接近生产级，有少量已知问题待修 |
| 10   | 满足所有生产级标准，无已知缺口 |

## 生产就绪阈值

**不要求所有维度达到10分**。同时满足以下全部条件即视为生产就绪，可停止迭代：
- 安全性 = 10 分（硬性要求，不满足不得上线）
- 稳定性 + 依赖健康度 均 ≥ 8 分
- 合规性与数据治理 ≥ 7 分（含 PII 处理）
- 其余维度均 ≥ 7 分
- 无任何维度低于 6 分

## 工作流程

### 第一步：读取或创建评分表

检查当前项目根目录是否存在 `VIBE_SCORECARD.md`。

**如果不存在**，先探索项目代码（读取关键文件，了解功能），然后判断项目类型并在评分表头部声明：

- **纯后端 API**：有路由/handler，无前端页面
- **前端**：主体是 UI 组件，无服务端逻辑
- **全栈**：同时含前端页面和后端 API
- **CLI 工具**：命令行入口，无 HTTP 服务

项目类型决定「开发者/用户体验」维度使用后端还是前端的10分标准。声明后创建评分表：

```markdown
# Vibe Scorecard

**项目类型**：[纯后端 API / 前端 / 全栈 / CLI 工具]

> ## 维度定义与生产级（10分）标准
>
> **功能完整性**
> 范围：需求覆盖、边缘案例处理、功能正确性
> 10分标准：先列出功能清单（逐条列出已实现/未实现），基于清单打分；空值/超限/并发边缘案例有测试覆盖；未实现条目数为0
> 检测命令：`grep -rn "TODO\|FIXME\|HACK\|XXX" --include="*.go" --include="*.ts" --include="*.py" .` 扫描遗留标记；对照功能清单逐条核查
>
> **安全性**
> 子维度：认证授权 / 输入验证 / 加密与secrets / OWASP合规
> 范围：认证授权、输入验证、数据加密、OWASP Top 10
> 10分标准：无 OWASP Top 10 漏洞；secrets 不在代码库（用环境变量/密钥管理）；敏感数据传输加密、存储脱敏；最小权限原则；有 SQL 注入/XSS 防护
> 检测命令：`semgrep --config=auto .`；`trivy fs .`；`grep -rn "password\|secret\|apikey\|api_key" --include="*.go" --include="*.ts" --include="*.py" .` 扫描硬编码凭证
>
> **稳定性**
> 范围：运行时稳定、错误处理、超时重试、优雅关闭
> 10分标准：所有外部调用（DB、HTTP、队列）有超时和重试；panic/未捕获异常有兜底；进程收到 SIGTERM 能优雅关闭；错误携带上下文（不裸 wrap）
> 检测命令：`grep -rn "http\.Get\|http\.Post\|sql\.Query" --include="*.go" .` 确认均传入 context；`grep -rn "recover()" --include="*.go" .` 检查 panic 兜底；`grep -rn "signal\.Notify\|SIGTERM" --include="*.go" .`
>
> **依赖健康度**
> 子维度：CVE扫描 / License合规 / 版本锁定 / CI自动扫描
> 范围：第三方库版本、已知 CVE、License 合规
> 10分标准：无已知高危 CVE（CVSS ≥7.0）；依赖版本锁定（lock 文件存在且提交）；所有 License 与项目兼容（无 GPL 污染商业项目）；CI 中有自动扫描
> 检测命令：`npm audit --audit-level=high` / `govulncheck ./...` / `pip-audit`；`npx license-checker --summary` / `go-licenses check ./...`；检查 package-lock.json / go.sum 是否存在
>
> **测试策略**
> 子维度：单测覆盖率 / 集成测试 / E2E / CI自动化
> 范围：单测/集成/E2E 分层、覆盖率、CI 自动化
> 10分标准：核心业务逻辑单测覆盖率 ≥80%；主要 API 接口有集成测试；CI 每次提交自动运行全部测试；测试本身可维护（无魔法数字、无重复 setup）
> 检测命令：`go test -coverprofile=coverage.out ./... && go tool cover -func=coverage.out | grep total` / `jest --coverage --coverageReporters=text-summary`；检查 .github/workflows/ 或 .gitlab-ci.yml 中的 test job
>
> **代码质量**
> 子维度：圈复杂度 / 重复率 / 命名规范 / 函数长度
> 范围：可读性、命名、重复代码、函数复杂度
> 10分标准：无重复逻辑（DRY）；函数单一职责、长度 <50 行；无单字母变量名（i/j 循环变量除外）、无 `data/tmp/result/utils/helper` 等歧义命名；圈复杂度 <10；无注释掉的废代码
> 检测命令：`gocyclo -over 10 .` / `eslint --max-warnings 0`；`jscpd --min-tokens 50 .` 检测重复代码；`grep -rn "//.*=" --include="*.go" .` 扫描注释掉的代码
>
> **架构成熟度**
> 范围：模块解耦、依赖方向、接口抽象、可扩展性
> 10分标准：模块间无循环依赖；依赖方向单向（高层不依赖低层实现）；接口与实现分离；新增功能不需修改核心模块；模块边界清晰可验证
> 检测命令：`go tool vet ./...`；`madge --circular src/`；`grep -rn "import" --include="*.go" .` 确认内部包未被跨层引用；`deptrac analyze`（PHP/Java）
>
> **性能**
> 子维度：P99响应时间 / N+1查询 / 索引 / 内存泄漏
> 范围：响应时间、吞吐量、数据库查询、资源使用
> 10分标准：P99 响应 <500ms（或符合业务 SLA）；无 N+1 查询；热点查询有索引；无内存泄漏；大数据量用分页/流式处理
> 检测命令：`EXPLAIN ANALYZE <慢查询>` 检查执行计划；`k6 run load_test.js` / `artillery run` 压测；Go: `go tool pprof`；JS: Chrome DevTools Memory snapshot
>
> **可观测性**
> 子维度：结构化日志 / Metrics / 链路追踪 / 告警规则
> 范围：日志、指标、链路追踪、告警
> 10分标准：结构化日志（JSON 格式含 level/time/trace_id）；关键业务操作有 metrics（请求量、错误率、延迟）；错误有 trace ID 可追查；至少2条告警规则（①服务不可用 ②错误率 >5% 持续1分钟）
> 检测命令：`grep -rn "fmt\.Print\|console\.log\|print(" --include="*.go" --include="*.ts" .` 检查非结构化日志；验证 /metrics 端点；检查 Prometheus rules 或 Grafana 告警配置文件
>
> **文档质量**
> 子维度：API文档 / README / ADR / 运维文档
> 范围：API 文档、README、架构决策记录
> 10分标准：API 有 OpenAPI/Swagger 或同等规格文档；README 让新人 10 分钟内能在本地跑起来；关键架构决策有 ADR 记录；部署/运维步骤有文档
> 检测命令：检查 swagger.yaml / openapi.json / docs/swagger/ 是否存在；逐行执行 README 中的 Quick Start 命令验证可用性；`ls docs/adr/` 检查 ADR 目录
>
> **开发者/用户体验**
> 范围（后端服务）：接口易用性、错误信息、本地开发体验
> 范围（前端/全栈）：核心流程顺畅度、错误提示、加载状态
> 10分标准（后端）：① 所有错误响应含 code+message+request_id ② 接口路径/命名风格统一（全 RESTful 或全 RPC，无混用）③ 本地一条命令启动完整环境（如 make dev）④ 无 500 错误直接暴露 stack trace
> 10分标准（前端）：关键流程无多余步骤；所有异步操作有加载态；错误有友好提示而非裸 stack trace
> 检测命令：`grep -rn "500\|InternalServerError\|stack\|stacktrace" --include="*.go" --include="*.ts" .`；curl 全部错误接口确认响应格式
>
> **合规性与数据治理**
> 子维度：PII识别与脱敏 / 数据保留策略 / 合规检查项
> 范围：PII 处理、数据保留、适用法规（GDPR/CCPA/SOC2）
> 10分标准：PII 字段（email/phone/身份证等）有识别、脱敏或加密处理；日志中无裸露 PII；数据保留/删除策略有文档和实现；适用的合规要求有逐条检查项
> 检测命令：`grep -rn "email\|phone\|ssn\|passport\|id_card\|credit_card" --include="*.log" .` 检查日志 PII；`grep -rn "email\|phone" --include="*.go" --include="*.ts" .` 确认字段有脱敏处理；检查隐私政策/合规文档
>
> **可运维性**
> 子维度：健康检查 / Runbook / 备份恢复 / 配置管理
> 范围：运维文档、备份恢复、配置管理、故障自愈
> 10分标准：有 /health 和 /ready 端点；有 Runbook（常见故障处理步骤）；数据库/关键数据有备份策略和恢复演练记录；配置变更无需重新部署（环境变量/配置中心）；基础设施变更有版本控制（IaC）
> 检测命令：`curl -f http://localhost:PORT/health`；`curl -f http://localhost:PORT/ready`；检查 docs/runbook.md 或 runbooks/ 目录；检查备份 cron job 配置；检查 terraform/ 或 k8s/ 目录

## 功能清单（首次运行时填写，功能完整性打分依据）

| # | 功能描述 | 状态 | 备注 |
|---|---------|------|------|

## 历史记录

```

**如果已存在**，读取它，继续迭代。

### 第二步：选择当前要改进的维度

找出分数最低且**未达到生产就绪阈值**的维度。如果并列，按优先级顺序：安全性 > 依赖健康度 > 合规性与数据治理 > 架构成熟度 > 功能完整性 > 稳定性 > 测试策略 > 代码质量 > 性能 > 可观测性 > 可运维性 > 文档质量 > 开发者/用户体验。

### 第三步：修复

专注修复当前维度的问题，每次只处理一个维度：

1. **评估改动量**：如果预计改动超过 20 个文件或 500 行，先将当前维度拆成子任务（如安全性拆为：输入验证、secrets 管理、加密三个子任务），逐个处理
2. **先运行该维度的检测命令**，用输出结果确定问题范围（每个维度的检测命令见维度定义，如 gocyclo、govulncheck、grep 扫描、覆盖率报告等）
3. 详细分析问题根因
4. 实施修复（修改代码）；**每完成一个子问题立即 commit**：
   ```bash
   git add -A && git commit -m "vibe: [维度名] 修复[子问题] — 简述"
   ```
5. 自测验收（重新运行检测命令，确认问题已消除）
6. 如果遇到报错，先修复报错再继续

### 第四步：更新评分表并提交

整个维度修复完成后：
1. 更新 `VIBE_SCORECARD.md`：
   - 更新该维度的分数
   - 更新剩余问题描述
   - 在历史记录中追加一条记录，格式：`- [日期] [维度] X分→Y分 — 简述改了什么 | 下次继续：[下一个待处理维度或子任务]`
2. 执行最终 git commit：
   ```bash
   git add -A && git commit -m "vibe: [维度名] 从X分提升到Y分 — 简述改了什么"
   ```
   如果项目还没有 git 仓库，先执行 `git init` 再 commit。

### 第五步：判断是否生产就绪

检查是否同时满足以下所有条件：
- 安全性 = 10 分
- 稳定性 + 依赖健康度 均 ≥ 8 分
- 合规性与数据治理 ≥ 7 分
- 其余维度均 ≥ 7 分
- 无任何维度低于 6 分

**满足** → 输出生产就绪报告，总结各维度最终得分和主要改进内容，标注仍有提升空间的维度

**未满足** → 回到第二步，继续处理得分最低且阻塞上线的维度

## 注意事项

- **每次只处理一个维度**，不要贪多
- **每完成一个子问题立即 commit**，减少中断丢失进度的风险
- **中断后续跑**：直接重新输入 `/vibe`，会自动读取 `VIBE_SCORECARD.md` 从历史记录的「下次继续」断点接着跑
- **每次修复后必须自测**，确认有效再更新分数
- **描述要具体**，不要写「更好看」，要写「把按钮宽度从80px改为120px」
- **遇到卡住**，把任务拆小，逐步执行
- **遇到报错**，先把报错信息分析清楚再修

## 开始

现在开始：检查项目根目录是否有 `VIBE_SCORECARD.md`，然后执行上述流程。

**重要：整个过程中不要询问「是否继续」「是否proceed」，直接执行，只在真正遇到无法判断的歧义时才暂停提问。**
